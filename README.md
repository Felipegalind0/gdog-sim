# gdog-sim

![gdog_v1](gdog_v1.gif)

Wheeled quadruped simulation built with [Genesis](https://github.com/Genesis-Embodied-AI/Genesis), with a remote-control backend and a lunar visual theme.

## Features

- Procedural robot generation from `gdog.urdf.jinja` on every run
- Seeded reproducibility via `--seed` (robot + terrain + visual randomization)
- Randomized multi-patch terrain (`Terrain` morph) with flat center spawn patch
- Space-style emissive sky sphere + moon-like terrain albedo
- Follow camera that tracks robot position/yaw with orbital state (`lon`, `lat`, `zoom`)
- Compact suspension HUD in the viewer (Pitch/Roll vertical stacks + suspension state)
- In-view text command console (`/` or `t`) for live tuning
- FastAPI control backend:
  - WebSocket endpoint: `/ws`
  - Optional WebRTC offer endpoint: `/offer`
  - Capability endpoint: `/capabilities`

## Requirements

- Python 3.10+ (3.11 recommended)
- macOS Apple Silicon works well with Genesis GPU backend
- A Python virtual environment (the repo `.venv` is recommended)

## Installation

Use the existing repo virtual environment first. Only create a new one when `.venv` does not exist.

Reuse existing `.venv`:

```bash
source .venv/bin/activate
python -m pip install -r requirements.txt

# Optional: enable WebRTC transport support
python -m pip install -r requirements-webrtc-optional.txt
```

Create `.venv` only if missing:

```bash
python3 -m venv .venv
source .venv/bin/activate
python -m pip install -r requirements.txt

# Optional: enable WebRTC transport support
python -m pip install -r requirements-webrtc-optional.txt
```

## Run

Interactive viewer:

```bash
python main.py --render
```

By default, `--render` now runs continuously until you close the viewer window or press `Ctrl+C`.

Interactive viewer with deterministic world generation:

```bash
python main.py --render --seed 12345
```

Optional step override (useful for short tests):

```bash
# Run exactly 2000 simulation steps
python main.py --render --steps 2000

# Unlimited loop explicitly
python main.py --render --steps 0
```

Headless video render:

```bash
python main.py --video
```

## Viewer Camera Controls

- Follows robot position/yaw automatically
- Mouse rotate drag adjusts camera latitude (`lat`)
- Mouse wheel:
  - horizontal scroll changes orbit longitude (`lon`)
  - vertical scroll changes latitude (`lat`)
- Hold `Shift` + wheel to zoom (`zoom`)

## Suspension HUD Legend

The HUD now has two layers:

- Always-on summary (top-center): suspension on/off and current pitch/roll angles.
- Optional debug panel (single spot at top-left): appears only when debug flags are enabled.

Debug flags:

- `debug_pitch=on|off`
- `debug_roll=on|off`
- `debug_yaw=on|off`
- `debug_speed=on|off`

When a debug flag is enabled, all relevant values for that algorithm are printed together in the same panel section.

Per-axis fields:

- `Kp` (proportional gain): reacts to current angle error. Higher values correct faster, but too high can oscillate.
- `Ki` (integral gain): accumulates long-term error to remove steady-state bias/drift.
- `Kd` (derivative gain): reacts to how fast the angle error is changing. Helps damping and reduces overshoot.
- `dz`: vertical leg-target correction generated by that axis controller (meters). Positive/negative direction depends on sign settings and joint conventions.

`Ki` visibility:

- `Ki` is shown directly in the panel when `debug_pitch` or `debug_roll` is on.
- `status` always shows full gain sets: `p i d` and `rp ri rd`.

Why do `p_sign` / `r_sign` exist?

- They are runtime polarity toggles (`+1` / `-1`) for pitch and roll correction mixing.
- They exist so you can flip correction direction instantly if axis conventions differ, without rewriting IK/PID math.
- They are not shown in the compact HUD; use `status`, `p_sign?`, and `r_sign?` when needed.

## Suspension Console Quick Commands

- Open console: press `/` or `t`
- Run command: `Enter`
- Cancel input: `Esc`
- Useful commands:
  - `help`
  - `status`
  - `p=0.10`, `d=0.02`, `i=0.00`
  - `rp=0.10`, `rd=0.02`, `ri=0.00`
  - `p_sign=1` or `p_sign=-1`
  - `r_sign=1` or `r_sign=-1`
  - `debug_pitch=on`, `debug_roll=on`
  - `debug_yaw=on`, `debug_speed=on`
  - `respawn`
  - `reset`

## Run With Web Controller

Run the simulator and remote UI in separate terminals.

1. Terminal A (this repo):

```bash
source .venv/bin/activate
python main.py --render
```

2. Terminal B (`../gdog-remote`):

```bash
npm install
npm run dev
```

3. Open the Vite URL (usually `http://localhost:5173`)

## Transport Behavior

- WebSocket is the default command path
- WebRTC is optional and only enabled when backend capability probe reports support
- If `aiortc` is missing, backend automatically stays in WebSocket mode

## Troubleshooting

- `aiortc not installed. WebRTC disabled. Using WebSockets as primary.`
  - This is expected unless you install optional WebRTC dependencies
- Frontend shows controls but robot does not move:
  - Confirm simulator is running
  - Confirm backend responds at `http://localhost:8000/capabilities`
- Startup/import issues:
  - Activate `.venv`
  - Reinstall with `pip install -r requirements.txt`
